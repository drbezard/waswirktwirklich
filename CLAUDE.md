# CLAUDE.md — Was Wirkt Wirklich

## Project Overview

Evidence-based medical patient articles (in German) — generated by AI, reviewed by medical specialists. The site presents health information backed by RCTs, Cochrane reviews, and meta-analyses, with full transparency about the AI prompts used.

**Live site:** https://waswirktwirklich.com

## Tech Stack

- **Astro 6** — Static site generator with Server Islands for the reviewer-verification badge (live from Supabase)
- **Tailwind CSS 4** — Utility-first styling via Vite plugin (`@tailwindcss/vite`)
- **TypeScript** — Strict mode (extends `astro/tsconfigs/strict`)
- **Markdown** — Article content via Astro Content Collections
- **Vercel** — Hosting, auto-deploys on push to `main`
- **Node.js >= 22.12.0** — Required runtime

## Architecture & Key Patterns

### Content Collections

Articles live in `src/content/artikel/` as Markdown files. The schema is defined in `src/content.config.ts` using Zod:

```typescript
// Required fields
title: string       // Article title
slug: string        // URL slug (used in /artikel/[slug])
date: string        // ISO date string (e.g. "2026-04-10")
category: string    // Medical specialty (e.g. "Orthopädie", "Innere Medizin")
excerpt: string     // Short description for cards and SEO

// Optional fields
image: string        // Hero image URL (Supabase Storage, bucket "article-images")
draft: boolean       // If true, article is not built/published (default: false)
prompt: string       // The AI prompt used (shown publicly for transparency)
sources: string[]    // Citation list
seoTitle: string
seoDescription: string
// Reviewer data (name, photo, title etc.) lives ONLY in Supabase (public.profiles
// + public.verifications) and is loaded live via the ReviewerBadge Server Island.
// Do NOT add reviewer* fields back to the Markdown frontmatter.
```

### Routing

- `/` — Homepage (latest 5 articles, client-side search, 3-pillar principle section)
- `/artikel/[slug]` — Individual article (rendered from Markdown)
- `/fachgebiete` — All medical specialties overview
- `/fachgebiet/[slug]` — Articles filtered by specialty (slug derived from category name)
- `/impressum` — Legal page
- `/rss.xml` — RSS feed for article subscriptions
- `/404` — Custom error page

### Styling

- **Theme colors** are defined in `src/styles/global.css` using OKLCH color space via `@theme {}` block
- **Primary color:** blue (`oklch(0.55 0.2 260)`)
- **Font:** Inter (Google Fonts), with SF Pro Display as display fallback
- **Custom CSS classes** used in article Markdown content:
  - `.studie` — Study citation box (white background, subtle shadow, border)
  - `.studie-name` — Study name heading (blue, bold)
  - `.studie-details` — Study metadata (journal, sample size)
  - `.kernaussage` — Key finding callout (blue left border, light blue background, auto-prefixed "KERNAUSSAGE" label)
- **Article content** uses `.article-content` class with prose overrides for typography
- **Layout:** max-width `4xl` (article/content) or `5xl` (card grids), centered with `mx-auto px-5`

### Search

Client-side search on the homepage:
- Article data serialized as inline JSON via `define:vars`
- Searches title, category, excerpt, and slug fields
- 200ms debounce on input, dropdown results panel
- No external search service

### Review System

Review/verification is dynamic, driven by the Supabase database:
- Source of truth: `public.verifications` + `public.profiles` (photo, name, title, bio, website)
- `src/components/ReviewerBadge.astro` is a Server Island (`server:defer`) that queries
  Supabase at request time and renders a "Geprüft von Dr. X" badge (variant `card` on
  lists, variant `full` on article detail pages).
- If no active, non-revoked, non-expired verification exists for a given slug → the
  island renders an "Ausstehend"-state.
- The doctor-facing flow for verifying is under `/arzt/artikel/[id]` and writes through
  the `verify_article` RPC to the database.

## Adding a New Article

1. Create a new `.md` file in `src/content/artikel/` (filename should match the slug)
2. Add required frontmatter:
   ```yaml
   ---
   title: "Article Title"
   slug: "article-slug"
   date: "2026-04-11"
   category: "Fachgebiet"
   excerpt: "Short description"
   draft: true   # set to false once reviewed by Manus + hero image added
   ---
   ```
3. Write article content in Markdown below the frontmatter
4. Use the correct HTML structure for study citation boxes and key finding callouts:
   ```html
   <!-- Study box (every tag must be closed!) -->
   <div class="studie">
   <span class="studie-name">Study Name: Author et al. (Year)</span>
   <div class="studie-details">Study type · <em>Journal Name</em> · N participants</div>

   Study description paragraph...

   </div>

   <!-- Key finding callout -->
   <section class="kernaussage">

   Summary text...

   </section>
   ```
5. Commit and push — Vercel deploys automatically

## Conventions

### Language

- All user-facing content is in **German**
- Code, comments, and commit messages may be in English or German
- Variable names and code identifiers are in English

### Commit Style

Recent commits use short, descriptive messages prefixed with context (e.g., "Fix:", "v5:", "Initial:"). Keep commit messages concise.

### Code Style

- Astro components use the standard `---` frontmatter fence for TypeScript logic
- Props interfaces are defined inline in component frontmatter
- Tailwind utility classes are used directly in templates (no separate CSS modules)
- Dates formatted with `toLocaleDateString('de-DE', ...)` for German locale
- Images are lazy-loaded (`loading="lazy"`)

### Images

- Article hero images are hosted in Supabase Storage, bucket `article-images`.
- Public URL pattern: `https://qyaivjcczncckifsrrps.supabase.co/storage/v1/object/public/article-images/{slug}.png`
- Doctor profile photos are hosted in Supabase Storage, bucket `reviewer-photos` (planned), URL stored in `public.profiles.photo_url`.
- Images are not stored in the repository.

## Testing

No automated test suite is configured. Verify changes by:
1. Running `npm run dev` and checking the browser
2. Running `npm run build` to ensure the static build succeeds

## Deployment

- **Platform:** Vercel (auto-detected Astro framework)
- **Trigger:** Automatic on push to `main`
- **No custom config:** No `vercel.json` — uses Vercel defaults
- **Output:** Mostly static; `/arzt/*`, `/admin/*`, `/auth/*` and Server Islands run on Vercel serverless functions
- **Pre-build hook:** `scripts/sync-articles.mjs` syncs Markdown article metadata into Supabase `articles` table before every `astro build` (both locally and on Vercel).
- **Sitemap:** Auto-generated via `@astrojs/sitemap` integration
- **RSS Feed:** Available at `/rss.xml` via `@astrojs/rss`

## Common Pitfalls

- The `<div class="studie">` blocks in article Markdown use raw HTML — ensure all tags are properly closed (`</span>`, `</div>`, `</section>`)
- Content Collections use the `glob` loader pattern — article filenames must end in `.md`
- The `searchData` on the homepage intentionally excludes `reviewerPhoto` to avoid bloating the inline JSON
- Tailwind CSS 4 uses the Vite plugin approach (`@tailwindcss/vite`), not the PostCSS plugin — configuration is in `global.css` via `@theme {}`, not in a `tailwind.config` file
